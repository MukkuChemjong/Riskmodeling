# Bankruptcy Risk Detection API

A machine learning API that predicts corporate bankruptcy likelihood using a Gradient Boosting Classifier trained on 95 financial ratio features. The system accepts financial data, scores it through a trained model, and returns a bankruptcy probability score with a risk band classification.

## Project Overview

| Property | Value |
|---|---|
| Model | Gradient Boosting Classifier |
| Dataset | Taiwan Corporate Bankruptcy (6,819 records, 95 features) |
| Target | `Bankrupt?` — 0 (Legitimate) or 1 (Bankrupt) |
| Class Imbalance | 30:1 (3.23% bankrupt) |
| Decision Threshold | 0.35 (adjusted for imbalance) |
| ROC-AUC (Test) | 0.9163 |
| Framework | Python 3.11 + FastAPI |
| Deployment | Docker + GCP Cloud Run |

## Docker Containerisation

### Step 1 — Verify Docker is Running

Docker Desktop must be open and fully started before running any Docker commands. The Docker whale icon in your system tray should be still (not animating). Verify with:

```bash
docker --version
```

If this returns an error about the Docker daemon not running, open Docker Desktop and wait for it to finish starting.

### Step 2 — Build the Docker Image

Run this from the project root directory where the Dockerfile is located:

```bash
docker build -t bankruptcy-api:v2.1.0 .
```

What this does: Docker reads the Dockerfile top to bottom, creates a Python 3.11 environment, installs all dependencies from requirements.txt, copies the application code and trained model files into the image, and produces a tagged image stored locally.

The first build takes several minutes because it downloads the base Python image and installs all dependencies. Subsequent builds reuse cached layers and complete in seconds if only application code has changed.

Verify the image was created:

```bash
docker images
```

You should see `bankruptcy-api` with tag `v2.1.0` in the list.

### Step 3 — Run the Container Locally

```bash
docker run -p 8080:8080 bankruptcy-api:v2.1.0
```

The `-p 8080:8080` flag maps port 8080 on your machine to port 8080 inside the container. The API is now accessible at `http://localhost:8080`.

To run in the background (detached mode):

```bash
docker run -d -p 8080:8080 --name risk-api bankruptcy-api:v2.1.0
```

### Step 4 — Verify the Container is Working

```bash
curl http://localhost:8080/api/v1/health
```

Expected response:

```json
{
  "status": "healthy",
  "model_version": "v2.1.0",
  "model_loaded": true,
  "uptime_seconds": 12
}
```

### Step 5 — Useful Docker Commands

```bash
# List running containers
docker ps

# View container logs
docker logs risk-api

# View logs in real time
docker logs -f risk-api

# Stop the container
docker stop risk-api

# Remove the container
docker rm risk-api

# Remove the image
docker rmi bankruptcy-api:v2.1.0

# Open a shell inside the running container (for debugging)
docker exec -it risk-api /bin/bash
```

## GCP Cloud Run Deployment

### Prerequisites

- Google Cloud CLI installed and on your PATH
- A GCP project created with billing enabled
- Your GCP project ID (find it at console.cloud.google.com)

Replace `YOUR_PROJECT_ID` with your actual GCP project ID in every command below.

---

### Step 1 — Authenticate with Google Cloud

```bash
gcloud auth login
```

This opens a browser window for you to sign in with your Google account. After signing in, return to the terminal.

Set your active project:

```bash
gcloud config set project YOUR_PROJECT_ID
```

Verify you are authenticated and pointing at the correct project:

```bash
gcloud config list
```

---

### Step 2 — Enable Required GCP APIs

These APIs are disabled by default and must be enabled before the relevant services can be used:

```bash
gcloud services enable run.googleapis.com
gcloud services enable artifactregistry.googleapis.com
gcloud services enable cloudbuild.googleapis.com
```

Each command may take up to 60 seconds to complete.

---

### Step 3 — Create an Artifact Registry Repository

Artifact Registry is GCP's private container registry where your Docker image will be stored:

```bash
gcloud artifacts repositories create risk-api \
  --repository-format=docker \
  --location=us-central1 \
  --description="Bankruptcy Risk API container images"
```

Verify the repository was created:

```bash
gcloud artifacts repositories list
```

---

### Step 4 — Configure Docker to Authenticate with GCP

This allows your local Docker installation to push images to GCP's registry:

```bash
gcloud auth configure-docker us-central1-docker.pkg.dev
```

When prompted to confirm changes to the Docker config, type `Y` and press Enter.

---

### Step 5 — Tag the Local Image with the GCP Registry Path

Docker needs the full registry path in the image tag to know where to push it:

```bash
docker tag bankruptcy-api:v2.1.0 \
  us-central1-docker.pkg.dev/YOUR_PROJECT_ID/risk-api/bankruptcy-api:v2.1.0
```

---

### Step 6 — Push the Image to Artifact Registry

```bash
docker push \
  us-central1-docker.pkg.dev/YOUR_PROJECT_ID/risk-api/bankruptcy-api:v2.1.0
```

Docker uploads each layer to GCP. The first push may take several minutes depending on your internet connection. Subsequent pushes reuse unchanged layers and complete much faster.

Verify the image is in the registry:

```bash
gcloud artifacts docker images list \
  us-central1-docker.pkg.dev/YOUR_PROJECT_ID/risk-api
```

---

### Step 7 — Deploy to Cloud Run

```bash
gcloud run deploy bankruptcy-risk-api \
  --image us-central1-docker.pkg.dev/YOUR_PROJECT_ID/risk-api/bankruptcy-api:v2.1.0 \
  --region us-central1 \
  --platform managed \
  --memory 512Mi \
  --cpu 1 \
  --min-instances 1 \
  --max-instances 10 \
  --port 8080 \
  --allow-unauthenticated \
  --set-env-vars MODEL_VERSION=v2.1.0
```

**What each flag does:**

| Flag | Value | Purpose |
|---|---|---|
| `--image` | Registry path + tag | Tells Cloud Run which image to deploy |
| `--region` | us-central1 | Where GCP runs your containers |
| `--platform managed` | managed | Fully managed — no infrastructure to configure |
| `--memory` | 512Mi | RAM per container instance |
| `--cpu` | 1 | CPU cores per instance |
| `--min-instances` | 1 | Always keep one warm instance to avoid cold starts |
| `--max-instances` | 10 | Maximum instances during traffic spikes |
| `--port` | 8080 | Port your container listens on |
| `--allow-unauthenticated` | — | Allows public access without GCP credentials |
| `--set-env-vars` | MODEL_VERSION | Environment variable passed into the container |

When prompted `Do you want to continue?` type `Y` and press Enter.

---

## API Endpoints

### POST /api/v1/predict

Submits financial ratio data and returns a bankruptcy probability score.

**Request body:**

```json
{
  "company_id": "CORP-48291",
  "debt_ratio": 0.72,
  "current_ratio": 1.2,
  "net_worth_assets": 0.28,
  "retained_earnings_to_assets": 0.15,
  "roa_c": 0.45,
  "borrowing_dependency": 0.65
}
```

**Response:**

```json
{
  "company_id": "CORP-48291",
  "bankruptcy_risk": 0.8734,
  "classification": "high_risk",
  "risk_band": "RED",
  "model_version": "v2.1.0",
  "response_timestamp": "2026-03-20T14:22:10Z"
}
```

**Risk Bands:**

| Band | Score Range | Classification | Action |
|---|---|---|---|
| GREEN | 0.00 – 0.34 | low_risk | Approve |
| AMBER | 0.35 – 0.64 | medium_risk | Review |
| RED | 0.65 – 1.00 | high_risk | Escalate |

## Model Information

| Parameter | Value | Reason |
|---|---|---|
| Algorithm | GradientBoostingClassifier | Best for tabular imbalanced financial data |
| n_estimators | 200 | Sufficient trees without overfitting |
| learning_rate | 0.05 | Slow learning with more trees for stability |
| max_depth | 4 | Shallow trees prevent memorisation |
| subsample | 0.8 | Stochastic boosting reduces variance |
| min_samples_leaf | 10 | Prevents fitting to individual outliers |
| Decision Threshold | 0.35 | Increases recall on the minority class |
| Class Weighting | 30:1 | Mirrors the actual class imbalance ratio |

## Redeployment — Updating the Model

When a new model version is trained, follow these steps to push it to production with zero downtime.

### Step 1 — Retrain with the new data

```bash
python train_model.py
```

### Step 2 — Update the version tag in your Dockerfile and requirements

Change `v2.1.0` to the new version number, for example `v2.2.0`.

### Step 3 — Rebuild the Docker image with the new version tag

```bash
docker build -t bankruptcy-api:v2.2.0 .
```

### Step 4 — Tag for GCP

```bash
docker tag bankruptcy-api:v2.2.0 \
  us-central1-docker.pkg.dev/YOUR_PROJECT_ID/risk-api/bankruptcy-api:v2.2.0
```

### Step 5 — Push to Artifact Registry

```bash
docker push \
  us-central1-docker.pkg.dev/YOUR_PROJECT_ID/risk-api/bankruptcy-api:v2.2.0
```

### Step 6 — Deploy the new version to Cloud Run

```bash
gcloud run deploy bankruptcy-risk-api \
  --image us-central1-docker.pkg.dev/YOUR_PROJECT_ID/risk-api/bankruptcy-api:v2.2.0 \
  --region us-central1 \
  --set-env-vars MODEL_VERSION=v2.2.0
```

Cloud Run performs a rolling update with zero downtime. New instances running v2.2.0 are started and health-checked before any traffic is shifted away from the v2.1.0 instances. If the new instances fail their health checks, Cloud Run automatically rolls back to v2.1.0 without any manual intervention.

### Step 7 — Verify the new version is live

```bash
curl https://YOUR_CLOUD_RUN_URL/api/v1/health
```

The response should show `"model_version": "v2.2.0"`.
